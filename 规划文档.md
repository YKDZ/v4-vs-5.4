## 1. 项目背景与目标

### 1.1 项目概述

构建一个**全栈术语库（Termbase）管理系统**，后端采用 Hono.js + Drizzle ORM + PostgreSQL，前端采用 Vue 3 + Vite + TypeScript。系统采用国际标准 **TBX（TermBase eXchange，ISO 30042）** 所定义的**概念导向（Concept-oriented）**数据模型，核心维护模式为 **Concept + TermEntry**。

### 1.2 核心目标

- 实现术语的**概念级别**管理：一个概念可关联多个语言中的多个术语（同义词）
- 实现**多语言术语**的创建、查询、更新、删除（CRUD）
- 实现**术语回归验证**机制：确保译文中的术语使用与术语库保持一致
- 支持术语的**导入导出**（TBX 格式）
- 提供简洁的 Vue 前端界面用于日常术语库交互

### 1.3 适用场景

- 翻译公司/本地化团队的术语管理
- 企业知识库中的受控词汇表管理
- 多语言内容创作中的术语一致性保障


## 2. 术语管理领域知识

### 2.1 TBX 标准核心概念

TBX（TermBase eXchange）是 ISO 30042 标准定义的术语数据交换格式，其核心特征包括：

- **概念导向（Concept Orientation）**：以概念为核心组织术语数据，而非以词汇为中心
- **多语言同义**：同一概念下可记录多种语言的多个同义术语
- **结构化元数据**：支持定义、词性、语境、学科领域等丰富的数据类别

### 2.2 Concept + TermEntry 模式

本系统遵循的层级结构（从上到下）：

```
Concept（概念层）
├── 管理信息（创建日期、修改日期、记录编号等）
├── 概念级信息（定义、解释、学科领域）
└── Language Section（语言层）
    ├── 语言代码（如 en, zh-CN, ja）
    └── TermEntry（术语条目层）
        ├── 术语文本
        ├── 词性
        ├── 术语类型（全称/缩写/俗称）
        ├── 语境示例
        └── 状态（首选/允许/禁用）
```

**层级映射关系**（来自商业系统 MultiTerm）：

| 商业系统层级 | TBX 层级 | 本系统模型 |
|-------------|---------|-----------|
| Concept level | termEntry | `concept`（概念） |
| Index level | langSet | `language_section`（语言段） |
| Term level | term | `term_entry`（术语条目） |

### 2.3 TBX 数据类别

TBX 的核心数据类别：

- 术语文本（term）
- 定义（definition）
- 词性（part of speech）
- 语境（context）
- 学科领域（subject field）
- 术语类型（term type：fullForm, acronym, abbreviation, variant）
- 状态（administrative status：preferred, admitted, deprecated）


## 3. 商业级术语库回归验证调查

### 3.1 术语回归（Term Regression）的定义

术语回归验证是指：在翻译或内容生产流程中，**自动检查译文/内容中使用的术语是否与术语库中定义的标准术语保持一致**。这是商业级术语管理系统的核心功能之一。

### 3.2 SDL Trados Studio + MultiTerm 的术语验证器

**SDL MultiTerm** 是全球最广泛使用的商业级术语管理系统，其与 **Trados Studio** 集成，提供术语验证器（Terminology Verifier）用于术语回归检查。

核心验证机制包括：

| 验证准则 | 说明 |
|---------|------|
| **目标术语使用检查**（Check for possible non-usage of the target terms） | 检查源文段中出现的已知术语，其对应译文段中是否使用了术语库中的标准翻译 |
| **禁用术语检查**（Check for terms which may have been set as forbidden） | 检查译文是否使用了术语库中被标记为"禁用"的术语 |
| **缺失目标术语检查**（Check for terms without target term equivalent） | 检查源文中的术语是否在术语库中缺少目标语言翻译 |
| **模糊匹配**（Minimum match value / Search depth） | 通过模糊匹配算法容忍术语的形态变化，可配置最小匹配值和搜索深度 |

### 3.3 术语回归的工作流程

商业级术语回归的典型工作流：

```
步骤 1: 加载术语库
步骤 2: 提取源文本中的候选术语（术语识别/Term Recognition）
步骤 3: 在术语库中查找匹配（模糊匹配 + 精确匹配）
步骤 4: 检查目标语言术语的使用情况
步骤 5: 报告验证结果（通过/警告/错误）
步骤 6: 生成术语回归报告
```

### 3.4 术语回归在本项目中的体现

本项目将实现一个**简化版术语回归验证模块**，包含：

1. **术语库快照**：基于现有术语库建立回归基准
2. **术语对比引擎**：输入待验证文本 → 提取术语 → 与术语库对比
3. **验证报告**：输出术语使用一致性报告（匹配/不匹配/缺失）
4. **术语质量评估**：参考 ISO/DIS 26162-4 标准评估术语数据质量（完整性、一致性、可靠性、准确性）

### 3.5 商业级产品对标

| 产品 | 术语回归方式 | 核心特点 |
|------|-------------|---------|
| SDL MultiTerm + Trados | Terminology Verifier 插件 | 模糊匹配、禁用术语检查、翻译记忆集成 |
| Wordbee | 内置术语验证 | 基于 TBX 3.0，支持自定义字段回归检查 |
| memoQ | QA 模块中的术语检查 | 集成翻译记忆和术语库统一验证 |
| RWS Terminology Verifier | 独立验证配置 | 支持云项目与本地项目的术语回归设置 |

### 4.2 关键文档摘要

#### 4.2.1 Drizzle ORM 核心概念

Drizzle ORM 是一个 TypeScript 优先的"无头"ORM，提供与原生 SQL 接近的 API。

**核心模块**：
- `drizzle-orm`：核心包，定义 schema 和查询
- `drizzle-orm/pg-core`：PostgreSQL 核心定义
- `drizzle-orm/node-postgres`：Node.js 环境下的 pg 驱动
- `drizzle-kit`：CLI 工具，用于生成和管理数据库迁移
- `drizzle-zod`：生成 Zod schema 用于验证

**Schema 定义方式**：
```typescript
import { pgTable, serial, text, integer } from 'drizzle-orm/pg-core';

export const users = pgTable('users', {
  id: serial('id').primaryKey(),
  name: text('name').notNull(),
  age: integer('age'),
});
```

**Relations（关系定义）**：
```typescript
import { relations } from 'drizzle-orm';

// 一对多
export const usersRelations = relations(users, ({ many }) => ({
  posts: many(posts),
}));

// 多对多
export const postsRelations = relations(posts, ({ one, many }) => ({
  author: one(users, { fields: [posts.authorId], references: [users.id] }),
}));
```

**Relational Queries（关系查询）**：
```typescript
// 使用 db.query 进行关系查询
const posts = await db.query.posts.findMany({
  with: {
    author: true,
    comments: {
      limit: 5,
      orderBy: (comments, { desc }) => [desc(comments.createdAt)],
    },
  },
});
```

#### 4.2.2 Hono.js 核心概念

Hono.js 是一个轻量级 Web 框架（Express 的精神继承者），支持任何 JavaScript 运行时。

**基本路由**：
```typescript
import { Hono } from 'hono';

const app = new Hono();

app.get('/', (c) => c.json({ message: 'Hello' }));
app.post('/items', async (c) => {
  const body = await c.req.json();
  return c.json({ created: body }, 201);
});
```

**验证中间件**（使用 Zod）：
```typescript
import { zValidator } from '@hono/zod-validator';
import { z } from 'zod';

const schema = z.object({ name: z.string() });
app.post('/items', zValidator('json', schema), (c) => {
  const data = c.req.valid('json'); // 类型安全的访问
  return c.json(data);
});
```

```
  index.ts        # 服务入口 & 路由注册
  routes/         # 路由定义
  client.ts       # 客户端（Hono RPC）
```

## 9. 术语回归验证模块

### 9.1 模块目标

实现术语回归验证引擎，对比输入文本与术语库，输出术语使用一致性报告。参考 SDL MultiTerm Terminology Verifier 和 ISO/DIS 26162-4 标准设计。

### 9.2 核心算法流程

```
输入：源文本、目标文本、源语言、目标语言
      ↓
1. 术语提取（Term Extraction）
   - 从源文本中基于术语库提取已知术语
   - 支持精确匹配 + 编辑距离模糊匹配
      ↓
2. 术语库查询（Termbase Lookup）
   - 查找每个提取术语在术语库中的目标语言对应术语
   - 获取 preferred 状态术语作为期望值
      ↓
3. 目标文本比对（Target Text Comparison）
   - 检查目标文本中是否包含期望术语
   - 模糊匹配容忍形态变化（如单复数、时态）
   - 检查是否使用了 deprecated 状态的术语
      ↓
4. 生成验证结果
   - exact_match: 译文使用了与术语库完全匹配的标准术语
   - fuzzy_match: 模糊匹配通过
   - no_match: 术语使用不一致
   - missing: 术语库缺少目标语言翻译
      ↓
5. 计算回归统计
   - 术语覆盖率
   - 一致性得分
   - 各状态分布
      ↓
输出：回归报告
```

### 9.3 模糊匹配实现

使用 **编辑距离（Levenshtein Distance）** 算法计算术语相似度：

```typescript
interface FuzzyMatchResult {
  matched: boolean;
  score: number;       // 0.0 - 1.0
  matchedTerm: string;
  matchType: 'exact' | 'fuzzy' | 'no_match';
}
```

- 精确匹配：score = 1.0
- 模糊匹配通过：score >= 配置阈值（默认 0.75）
- 匹配失败：score < 阈值
